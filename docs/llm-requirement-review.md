## 二、逐模块分析

### 模块 A：智能对话引擎

#### ✅ 合理之处

1. **意图分类**（8种 intent）覆盖了核心操作场景，不多不少
2. **槽位填充 + 多轮对话**的设计很务实，必填/可选/推断三层策略清晰
3. **自动决策引擎的置信度阈值**（>80% 自动 / 否则询问）是关键的体验保障
4. **主动询问策略**的 5 种场景定义精准，特别是"冲突检测"和"优先级争议"

#### ⚠️ 需要校准的点

| 问题 | 风险 | 建议 |
|------|------|------|
| **意图识别依赖 LLM 而非本地模型** | 每次输入都要调 API，延迟 500ms-2s，成本不可忽略 | 增加本地规则预筛层：先用 `parseInput.js` 做快速匹配，仅模糊输入才走 LLM。已有 `parseInput.js` 是好基础 |
| **多轮对话状态机未定义具体状态** | 文档只提了概念，缺少状态图 | Phase 1 先简化为 **线性追问流**（最多 3 轮），不做复杂分支。如：`WAITING_TITLE → WAITING_TIME → WAITING_CONFIRM` |
| **自动决策的"历史同类任务"统计** | 冷启动时没有历史数据 | 用 LLM 做首次推断（带 `temperature=0.1`），积累数据后切换到统计推断 |
| **对话面板 UI 未定义** | 和现有 CreateView 如何共存？ | 建议：CreateView 增加一个"AI 模式"切换按钮，点击后 textarea 变成对话气泡界面，复用现有布局 |

#### 📊 实现难度

| 子模块 | 难度 | 工时 | 依赖 |
|--------|------|------|------|
| 意图识别 Prompt | 🟢 低 | 0.5d | 无 |
| 槽位填充 + 追问 | 🟡 中 | 1.5d | 意图识别 |
| 多轮状态机 | 🟡 中 | 1d | 槽位填充 |
| 自动决策引擎 | 🔴 高 | 1.5d | 历史统计 + 上下文构建 |
| 对话 UI | 🟡 中 | 1d | React 组件 |

**Phase 1 实际工时估算：3-4 天**（文档估计 2-3 天，略偏乐观）

### 模块 B：每日播报

#### ✅ 合理之处

1. **播报内容模板**信息密度适中，不冗长不简陋
2. **三重触发方式**（自动/语音/手动）灵活
3. **"上周回顾"** 跨时间维度的对比很有价值

#### ⚠️ 需要校准的点

| 问题 | 风险 | 建议 |
|------|------|------|
| **"打开应用自动播报"体验可能干扰** | 用户打开应用可能是为了快速创建任务，弹出长篇播报反而打断心流 | 改为**非侵入式**：在日历视图顶部显示一个可折叠的"今日概览"卡片，默认折叠，点开才看详情 |
| **播报内容依赖 LLM 生成** | 每天早上都调一次 LLM，成本累积。且 LLM 生成"建议"质量不稳定 | 数据部分（今日任务、统计）用 SQL 直接查，只把"建议"和"个性化文案"交给 LLM。**混合模式**：70% 模板 + 30% LLM |
| **"上周回顾"的数据窗口** | 周一和周日的数据意义不同 | 周一时显示"上周回顾"，其他日子显示"最近 7 天" |

#### 📊 实现难度

| 子模块 | 难度 | 工时 | 说明 |
|--------|------|------|------|
| 数据查询层 | 🟢 低 | 0.5d | 纯 SQL，和现有 task:getAll 复用 |
| 播报 Prompt | 🟢 低 | 0.5d | 模板化 Prompt |
| 播报 UI 组件 | 🟡 中 | 1d | 可折叠卡片，需要设计 |
| 自动触发逻辑 | 🟢 低 | 0.5d | 首次打开检测 + 手动触发 |

**Phase 3 前半部分工时：2-2.5 天**

---

### 模块 C：每日复盘

#### ✅ 合理之处

1. **完成率统计 + 未完成项追踪**是核心价值
2. **时间分布分析**（会议/coding/沟通）如能实现会很有洞察力
3. **"连续完成天数"** 游戏化设计增加粘性

#### ⚠️ 需要校准的点

| 问题 | 风险 | 建议 |
|------|------|------|
| **"时间分布"分析依赖标签体系** | 如果用户没有给任务打标签（会议/coding/沟通），分析无法进行 | Phase 3 先做**无分类版本**（仅完成率），标签分类作为 Phase 4+ 的可选增强 |
| **"明日预览"** 和每日播报重复 | 早上的播报已经显示了今日任务，复盘中的明日预览是冗余的 | 复盘中去掉"明日预览"，改为"明天建议"——基于今日未完成项的建议 |
| **定时自动弹窗** | 如果用户在开会/投屏，弹窗是灾难 | 只做**手动触发 + 通知提醒**（系统通知："今天的复盘已生成，点击查看"），不做强制弹窗 |

#### 📊 实现难度

| 子模块 | 难度 | 工时 | 说明 |
|--------|------|------|------|
| 复盘数据查询 | 🟢 低 | 0.5d | SQL 聚合查询 |
| 复盘 Prompt | 🟡 中 | 0.5d | 需要好的总结能力 |
| 复盘 UI 组件 | 🟡 中 | 1d | 卡片组件 |
| 通知提醒 | 🟢 低 | 0.5d | Electron Notification API |

**Phase 3 后半部分工时：2-2.5 天**

---

### 模块 D：上下文记忆系统

#### ✅ 合理之处

1. **五种记忆类型**的分类很全面
2. **更新频率分级**（实时 / 每次交互 / 每周）务实

#### ⚠️ 需要校准的点

| 问题 | 风险 | 建议 |
|------|------|------|
| **"人物关系"推断过度设计** | 对个人效率工具来说，自动推断团队关系价值有限，且准确率低 | Phase 2 先不做人物关系推断，只做基础的**人名提取 + 存储** |
| **对话历史存储策略** | 全文存储会快速膨胀，且隐私风险高 | 只存储最近 **50 轮对话**，超过的用 LLM 生成摘要替换。不存储完整对话原文 |
| **`user_profile` 表字段太静态** | `work_hours_start/end` 手动设置很鸡肋 | 改为**自动学习**：分析 tasks 表中最常出现的时间段，自动推断工作时间。用户可手动覆盖 |

#### 📊 实现难度

| 子模块 | 难度 | 工时 | 说明 |
|--------|------|------|------|
| 历史统计服务 | 🟡 中 | 1d | SQL 聚合 + 缓存 |
| 上下文构建器 | 🔴 高 | 1.5d | 需要设计注入策略 + token 预算管理 |
| ai_decisions 记录 | 🟢 低 | 0.5d | 纯数据记录 |
| user_profile 自动学习 | 🟡 中 | 1d | 统计分析 |

**Phase 2 实际工时估算：3-4 天**（文档估计 2 天，明显偏乐观）

## 三、架构层面的关键问题

### 3.1 当前代码库的就绪度分析

基于对 `src/` 目录的实际审查：

| 能力 | 就绪度 | 说明 |
|------|--------|------|
| 数据库层 | 🟢 90% | sql.js 已就绪，只需加 3 张表 |
| IPC 通信 | 🟢 85% | 模式成熟，新增 channel 模板化 |
| LLM 调用 | 🟢 80% | `testModelConnection` 已验证多厂商兼容 |
| 语音输入 | 🟢 70% | Whisper 集成已可用，但仍在完善中 |
| 前端组件 | 🟡 60% | 需要新增 AI 对话面板组件 |
| 状态管理 | 🔴 30% | **没有全局状态管理**（无 Redux/Zustand），AI 对话状态需要跨组件共享 |

### 3.2 最大的架构风险：缺少状态管理

当前 App.jsx 只有简单的 `useState`，AI 对话引擎引入后会出现：
- 对话状态需要在 CreateView、Sidebar、独立的 ChatPanel 之间共享
- "AI 正在思考"的全局 loading 状态
- 对话历史需要在多个视图访问

**建议：Phase 1 开始前，先引入轻量状态管理。** 推荐 **Zustand**（1KB，无 boilerplate）：

```js
// stores/aiStore.js
const useAiStore = create((set) => ({
  isChatting: false,
  messages: [],
  currentIntent: null,
  pendingSlots: {},
  addMessage: (msg) => set(s => ({ messages: [...s.messages, msg] })),
  setIntent: (intent) => set({ currentIntent: intent }),
  // ...
}));
```

### 3.3 Token 成本控制

这是文档完全忽略的关键问题。以一个典型对话为例：

| 操作 | 输入 Token | 输出 Token | 预估成本(DeepSeek) |
|------|-----------|------------|-------------------|
| 意图识别 | ~200 | ~100 | ¥0.0004 |
| 槽位填充 + 追问 | ~500 | ~200 | ¥0.001 |
| 每日播报 | ~800 | ~400 | ¥0.0016 |
| 每日复盘 | ~1000 | ~500 | ¥0.002 |
| 自由对话(10轮) | ~2000 | ~1000 | ¥0.004 |

**日预估成本：¥0.01-0.05**（正常使用），月成本 ¥0.3-1.5。成本可控，但需要在 `usage_logs` 中精确记录。

### 3.4 流式输出的实现路径

文档 Phase 4 提到 SSE/streaming，但 Electron 的 IPC 不适合做真正的 streaming。两种方案：

| 方案 | 优点 | 缺点 | 推荐 |
|------|------|------|------|
| **方案 A：Main process 直连 Renderer** | 真正的流式 | 需要额外 WebSocket/SSE 端口，架构复杂 | ❌ |
| **方案 B：分段 IPC 推送** | 复用现有 IPC | 每次 `ai:chatStream` 推送一个 chunk | ✅ |

**推荐方案 B**：Main process 用 `res.on('data')` 接收流，每 50ms 通过 `webContents.send('ai:chatStream', chunk)` 推给 Renderer。体验接近真流式。

## 五、预期效果 vs 实际效果

### 会有惊喜的地方

| 场景 | 预期 | 实际 | 说明 |
|------|------|------|------|
| **"下午3点开会"** → 自动创建 | ✅ 流畅 | ✅ 甚至更好 | 追问补全信息后创建，比手动填表快 3-5 倍 |
| **每日播报** | ✅ 有用 | ✅ 粘性功能 | 如果做成非侵入卡片，会是打开应用的主要动力 |
| **自动优先级推断** | ✅ 省心 | ⚠️ 80% 准确 | 初期准确率不会太高，需要决策记录反馈闭环 |

### 可能低于预期的地方

| 场景 | 预期 | 实际风险 | 建议 |
|------|------|----------|------|
| **"像人类一样交流"** | 自由对话 | ❌ LLM 对话能力有限，深聊会暴露 AI 感 | 限定对话域：聚焦任务管理，不要试图做通用聊天机器人 |
| **"自动决策"** | AI 完全自主 | ⚠️ 用户对自动决策的容忍度因人而异 | 每个自动决策都要有"为什么"的解释 + 一键修改入口 |
| **"复盘洞察"** | 深度分析 | ⚠️ LLM 对任务语义理解有限，"建议"容易空洞 | 降低预期：复盘的核心价值是**数据聚合**（完成率、时间分布），而非 AI 建议 |
| **"上下文记忆"** | AI 越来越懂你 | ⚠️ 初期数据少，记忆效果有限 | 冷启动期接受"笨拙"，2-4 周后逐步改善 |

### 一个核心洞察

> **用户真正需要的是"减少操作步骤"，而不是"和 AI 聊天"。**
>
> 最能提升体验的不是对话本身，而是：
> 1. 语音输入一句话 → 自动创建任务（3 步变 1 步）
> 2. 早上打开应用 → 看到今日概览（0 步变 0 步，信息主动呈现）
> 3. 创建任务时 → 优先级/标签/时长自动填好（省去手动选择）
>
> 这三点做好了，即使没有"自由对话"，用户也会觉得产品"很智能"。

## 六、待确认问题的回答

文档中列了 4 个待确认问题，逐一回答：

| 问题 | 建议 |
|------|------|
| **1. 默认对话模型** | 让用户在模型管理页设置一个"默认对话模型"，同时保留场景绑定覆盖。优先级：场景绑定 > 默认模型 |
| **2. 播报触发时机** | **手动触发 + 首次打开静默生成**。打开应用时在顶部显示"今日概览已生成"提示，用户点击展开。不做自动弹窗 |
| **3. 语音交互深度** | Phase 1-3 只做"语音转文字"（已有 Whisper），不做"纯语音对话"。纯语音对话需要 TTS + VAD + 打断检测，工作量翻倍 |
| **4. 多设备同步** | 不做。个人效率工具的定位决定单设备足够。如果需要，以后用文件同步（iCloud/OneDrive）覆盖 db 文件即可 |

## 八、总结

### 做对了什么
1. "对话优先"的核心理念抓住了个人效率工具的正确方向
2. 四个模块的划分清晰，各自独立可验证
3. 优雅降级、隐私优先的设计原则务实
4. 风险识别和缓解策略到位

### 需要调整什么
1. **Phase 0 必须加**：状态管理 + 数据库迁移，否则后面会乱
2. **工时翻倍预期**：原 9 天 → 实际 12-16 天
3. **校准"对话"预期**：用户要的是"少操作"而非"多聊天"
4. **播报/复盘不做弹窗**：非侵入式卡片 + 通知提醒
5. **自由对话优先级降低**：Phase 4 可以无限延期，核心闭环在 Phase 1-3

### 一句话建议

> **先做 Phase 1（AI 创建任务），验证对话引擎闭环。这是整个系统的"心跳"——心跳稳了，其他模块都是锦上添花。**
