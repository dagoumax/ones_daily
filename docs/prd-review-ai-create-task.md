## 二、功能范围评审

### 🔴 评审意见 #1：US-3 "批量创建任务" 应移除 Phase 1

**问题**：用户故事 US-3（"一次对话中连续创建多个任务"）在功能清单 F1-F9 中完全没有对应的实现描述。这会导致 PRD 范围不闭合。

**建议**：US-3 移到 Phase 2 或标记为 `Out of Scope (Phase 1)`。Phase 1 只做**单任务创建**。

**影响**：无，只是文档修正。

---

### 🔴 评审意见 #2：F6 "用户回复修改" 的范围需要明确

**问题**：AC-6 说"用户可以修改优先级、时间等字段"，但 PRD 没有定义修改的交互方式——用户是**自然语言说"改成下午4点"**，还是**点击确认卡片上的编辑按钮**？

**分析**：自然语言修改需要再调一次 LLM 来重新解析意图和槽位，这引入了复杂度：
- 需要区分"确认"和"修改"两种用户意图
- 修改后需要重新走槽位填充 + 确认流程
- 可能导致循环（修改 → 确认 → 再修改 → 再确认...）

**建议**：Phase 1 采用**混合方式**：
- 确认卡片上每个字段旁边有**行内编辑按钮**（点击直接修改，不走 LLM）
- 用户也可以自然语言说"改成下午4点"，走 LLM 重新解析
- 但自然语言修改**不计入追问轮数**，避免用户因反复修改触发 3 轮回退

---

### 🟡 评审意见 #3：本地预筛的"有效"定义需量化

**问题**：F2 说"parseInput() 返回有效的 title 和 startTime 则直接创建"，但什么算"有效"没有定义。

**建议**：增加判定规则：
```
有效 title  = title.length ≥ 2 && title 不全是标点/数字
有效 startTime = startTime ≠ null && startTime > now
```

同时注意一个场景：用户输入"明天下午3点"——parseInput 能解析出 startTime 但 title 为空。PRD 已覆盖（"返回 title 但无 startTime"应改为"有 startTime 但无 title"也要追问标题）。

---

### 🟢 评审意见 #4：确认 F4 槽位的 scope

PRD 中 F4 列了 8 个槽位，Phase 1 实际能实现的有：

| 槽位 | Phase 1 | 说明 |
|------|---------|------|
| title | ✅ | LLM 提取 |
| start_time | ✅ | LLM 提取 |
| end_time | ⚠️ | 仅默认 +1h，不做历史推断（那是 Phase 2） |
| priority | ⚠️ | 仅关键词匹配（#P1）+ LLM 语义推断，不做历史统计推断 |
| tags | ⚠️ | 仅 LLM 语义提取，不做历史共现分析 |
| participants | ✅ | LLM 提取 |
| location | ✅ | LLM 提取 |
| notes | ⚠️ | LLM 生成建议质量不稳定，设为"可选展示" |

**结论**：范围合理，但需要在 F4 的表格中标注哪些是"Phase 1 简化版"，避免开发时产生歧义。

## 四、Prompt 模板评审

这是本次评审的核心关注点。

### 🔴 评审意见 #5：意图识别和槽位填充应该合并为一次 LLM 调用

**问题**：PRD 将意图识别（F3）和槽位填充（F4）设计为两次独立的 LLM 调用。这意味着：
- 每次创建任务至少 2 次 API 调用（意图识别 + 槽位填充）
- 如果追问 2 轮 = 2×2 = 4 次调用
- 延迟翻倍，成本翻倍

**分析**：对于 `create_task` 这个核心场景，意图识别和槽位提取可以用**同一个 Prompt** 完成：

```json
// 合并后的输出格式
{
  "intent": "create_task",
  "confidence": 0.95,
  "slots": { ... },
  "missing_fields": [],
  "follow_up_question": null
}
```

- 如果 intent ≠ create_task → 直接返回提示信息（不需要槽位）
- 如果 intent = create_task → 槽位已经在同一个响应里了

**建议**：合并为一次调用。保留独立的意图识别 Prompt 仅用于未来 Phase 4 的自由对话场景（用户可能说"今天怎么样"触发播报而非创建任务）。

**影响**：延迟减半，成本减半，代码简化。

---

### 🟡 评审意见 #6：Prompt 缺少 current_time 注入和时区处理

**问题**：Prompt 8.1（意图识别）没有注入当前时间，但时间解析需要知道"现在是什么时候"才能正确理解"明天下午"。

**建议**：两个 Prompt 都需要在 system message 中注入：
```
当前时间：2026-07-07T14:30:00+08:00（星期三）
```

同时，LLM 返回的 ISO 8601 时间需要后处理：用本地时区验证，如果 LLM 返回 UTC 时间则自动转换。

---

### 🟡 评审意见 #7：Prompt 缺少 negative examples 和 edge cases

**问题**：Prompt 只有正向规则，没有告诉 LLM"不要做什么"。

**建议**：在槽位填充 Prompt 中增加：

```
负面示例（不要这样做）：
❌ 输入"下午开会" → 不要编造具体时间（如14:00），应该标记 missing_fields: ["start_time"]
❌ 输入"买菜" → 不要编造地点，location 留空
❌ 输入"嗯" → 意图应为 unknown

边界处理：
- "明天"指下一个日历日，不是24小时后
- "下周"指下周一所在的周
- "晚上"默认19:00，"早上"默认09:00
- 如果用户输入包含多个任务（"开会然后吃饭"），只提取第一个
```

---

### 🟢 Prompt 结构评审：格式正确

Prompt 8.2 的 JSON 输出结构设计良好：
- `intent` + `confidence` 分离了分类和置信度
- `missing_fields` 数组让前端可以精确知道该追问什么
- `follow_up_question` 让 LLM 自己生成自然的追问文案

一个细节：`response_format: { type: "json_object" }` 只在 OpenAI/DeepSeek 新版 API 中支持。如果用旧版 API 或 Anthropic，需要在 Prompt 末尾强调"只返回 JSON，不要其他内容"。

## 六、组件设计评审

### 🟡 简化建议

PRD 设计了 4 个新组件，但 CreateView 本身已经有 480 行代码。如果全部内联到 CreateView 中，文件会膨胀到 800+ 行。

**建议**：将 AI 模式相关逻辑抽取为一个独立的大组件 `AICreatePanel.jsx`，CreateView 只负责模式切换：

```jsx
// CreateView.jsx 简化后
{chatMode ? (
  <AICreatePanel onCreated={handleTaskCreated} onCancel={handleCancel} />
) : (
  <>{/* 现有的手动模式 textarea + 预览 */}</>
)}
```

AIChatMessage / AIConfirmCard / AIInputBar 作为 AICreatePanel 的子组件，不需要单独暴露。

## 八、修订后的功能范围（Phase 1）

基于评审结论，Phase 1 明确范围为：

### ✅ 包含

- AI 模式切换入口（F1）
- 本地预筛层：明确输入直接创建（F2）
- 合并的意图识别 + 槽位填充（一次 LLM 调用）
- 线性多轮追问状态机（最多 3 轮，含回退）
- 确认卡片：展示 + 行内编辑 + 确认/取消（F6）
- 确认阶段支持自然语言修改（如"改成下午4点"）
- 优雅降级：API 失败 / JSON 异常 / 未配置模型（F8）
- 对话历史持久化（F9）
- 语音输入 → AI 解析（AC-13）

### ❌ 不包含（延至 Phase 2+）

- 批量创建任务（US-3）
- 历史数据驱动的自动决策（优先级/时长/标签基于统计）
- query_tasks / update_task / delete_task 意图（返回占位提示）
- 流式输出（Phase 4）
- 自由对话面板（Phase 4）
