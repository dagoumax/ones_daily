# LLM 深度需求分析 — AI 驱动的个人效率助手

> 版本：v0.2 | 日期：2026.07.07 | 状态：已审查，Phase 0 已就绪

---

## 一、核心理念

**AI 不是工具的一个功能，而是交互的第一界面。**

用户通过自然语言（文字/语音）与 AI 对话，AI 理解意图、补全信息、执行操作、主动汇报。传统的表单/按钮交互降级为"AI 无法处理时的 fallback"。

> **关键洞察**：用户真正需要的是"减少操作步骤"，而不是"和 AI 聊天"。最能提升体验的是：
> 1. 语音/文字一句话 → 自动创建任务（3 步变 1 步）
> 2. 打开应用 → 看到今日概览（信息主动呈现）
> 3. 创建任务时 → 优先级/标签/时长自动填好（省去手动选择）

### 设计原则

1. **对话优先** — 用户说一句话就能完成操作，不需要填表单
2. **主动智能** — AI 在合适的时间主动发起对话（播报、提醒、复盘），但**不做强制弹窗**，采用非侵入式卡片 + 通知提醒
3. **持续记忆** — AI 记住用户偏好、历史决策、常见模式（冷启动期接受"笨拙"，2-4 周逐步改善）
4. **优雅降级** — AI 不可用时，回退到现有 `parseInput.js` 规则解析 + 手动表单
5. **隐私优先** — 所有数据本地存储，LLM 调用仅发送必要上下文；对话历史只存最近 50 轮，超出自动压缩为摘要

## 三、功能体系

### 模块 A：智能对话引擎（核心）

```
┌──────────────────────────────────────────────────────┐
│                   对话管理                            │
│  ┌──────────┐  ┌──────────┐  ┌──────────────────┐   │

│  │ (parseInput)│ │ (LLM)    │  │ (LLM + 追问)     │   │
│  └──────────┘  └──────────┘  └──────────────────┘   │
│  ┌──────────┐  ┌──────────┐  ┌──────────────────┐   │
│  │ 上下文    │  │ 自动决策  │  │ 主动询问策略      │   │
│  │ context  │  │ auto     │  │ proactive        │   │
│  └──────────┘  └──────────┘  └──────────────────┘   │
└──────────────────────────────────────────────────────┘
```

#### A0. 本地预筛层（新增）

**在调 LLM 之前，先用 `parseInput.js` 做快速匹配。** 如果输入包含明确的日期时间 + 标题，直接本地解析，零延迟、零成本。

| 条件 | 处理 |
|------|------|
| 输入含明确时间 + 非空标题 | 本地 `parseInput()` 解析，直接创建 |
| 输入含明确时间但无标题 | 本地解析时间，追问标题（1 轮） |
| 输入模糊、歧义、或用户主动触发 AI | 走 LLM 意图识别 → 槽位填充 |

#### A1. 意图识别

| 意图 | 示例输入 | 目标操作 |
|------|----------|----------|
| `create_task` | "明天下午开会"、"帮我记一下买牛奶" | 创建任务 |
| `query_tasks` | "今天有什么安排"、"本周完成了什么" | 查询任务 |
| `update_task` | "把那个会议改到后天"、"标记完成" | 修改任务 |
| `delete_task` | "取消明天的会议" | 删除任务 |
| `daily_brief` | "早上好"、"今天怎么样" | 触发每日播报 |
| `daily_review` | "总结一下"、"今天做了什么" | 触发每日复盘 |
| `chat` | "你觉得我应该先做哪个" | 自由对话/建议 |
| `unknown` | 无法识别 | 追问澄清 |

**Phase 1 优先实现 `create_task` 意图**，其他意图在 Phase 2-4 逐步实现。

#### A2. 槽位填充 + 多轮对话

任务创建的必填/可选槽位：

| 槽位 | 必填 | 推断策略 |
|------|------|----------|
| title | ✅ | 从输入中提取核心动作 |
| start_time | ✅ | 从输入解析 + 默认当前时间后最近整点/半点 |
| end_time | ❌ | 历史同类任务平均时长推断（冷启动默认 1h） |
| priority | ❌ | 关键词 + 历史模式 + 截止日期紧迫度 |
| tags | ❌ | 历史标签频率 Top3 + 内容语义 |
| participants | ❌ | 从输入提取人名（不做团队关系推断） |
| location | ❌ | 从输入提取地点 |
| notes | ❌ | AI 补充建议（准备材料、前置任务等） |

**多轮对话状态机**（Phase 1 简化为线性追问，最多 3 轮）：

```
IDLE → WAITING_TITLE → WAITING_TIME → WAITING_CONFIRM
```

**多轮对话流程示例**：

```
用户: "下午开个会"
  → 意图: create_task
  → 槽位: title="开会", start_time=今天下午(模糊)
  → 缺失: 具体时间、主题、参与人

AI: "好的，帮你创建会议。请问：
     1️⃣ 具体几点开始？
     2️⃣ 会议主题是什么？
     3️⃣ 有哪些人参加？"

用户: "3点，讨论Q3产品规划，叫上产品组"
  → 槽位: start_time=15:00, title="Q3产品规划讨论", participants=["产品组"]
  → 推断: priority=P1（历史"产品规划"类任务多为P1）
           tags=["产品","会议","Q3"]
           end_time=16:30（历史会议平均1.5小时）

AI: "已创建：
     📋 Q3产品规划讨论
     🕐 今天 15:00-16:30
     🔴 P1 重要
     👥 产品组
     #产品 #会议 #Q3
     
     建议：会前回顾上次产品评审结论。需要我帮你调出来吗？"
```

#### A3. 自动决策引擎

AI 在以下情况**自动决策**（不询问用户）：

| 决策点 | 规则 | 置信度要求 |
|--------|------|-----------|
| 优先级推断 | 历史同类任务优先级分布 + 关键词匹配 + 截止日期紧迫度 | >80% 自动，否则询问 |
| 时长推断 | 历史同类任务平均时长（最近10次）；冷启动默认 1h | 直接应用 |
| 标签推荐 | 历史标签共现频率 Top3 | 直接应用 |
| 默认时间 | 未指定时间 → 当前时间后最近整点/半点 | 直接应用 |

每个自动决策都要附带 **"为什么"的解释** + 一键修改入口。

#### A4. 主动询问策略

AI **必须询问**的情况：

| 场景 | 示例 |
|------|------|
| 必填信息缺失 | 没有时间、没有标题 |
| 存在歧义 | "明天" vs "下周一"（跨周末） |
| 冲突检测 | 新任务时间与已有任务重叠 |
| 优先级争议 | 用户说 P3 但关键词暗示紧急 |
| 高风险操作 | 删除、批量修改 |

### 模块 B：每日播报（Morning Brief）

**触发方式**（修订）：
- ~~用户首次打开应用（每天一次）~~ → **首次打开静默生成**，顶部显示"今日概览已生成"可折叠卡片
- 用户说"早上好"/"今天有什么" → 展开播报
- 手动点击播报按钮

**实现策略**：**混合模式** — 70% SQL 模板查询 + 30% LLM 生成建议

| 内容 | 来源 |
|------|------|
| 今日待办列表 | SQL 直接查 `tasks` 表 |
| 上周完成统计 | SQL 聚合查询 |
| 个性化建议 | LLM 基于任务列表生成 |
| 天气信息 | Phase 4 可选集成 |

**播报内容模板**：

```

📋 今日待办 (3项)：

  ✅ 完成 12 项
```

### 4.2 模型路由策略

| 任务类型 | 推荐模型 | 原因 |
|----------|----------|------|
| 意图识别 + 槽位填充 | DeepSeek V4 Flash / GPT-5.4 Mini | 结构化输出，低成本 |
| 每日播报/复盘 | DeepSeek V4 Pro / GPT-5.5 | 需要深度推理和总结 |
| 自由对话 | 用户配置的默认模型 | 用户偏好 |
| 自动决策（优先级等） | 规则引擎 + LLM fallback | 能规则解决的不用模型 |

### 4.3 Token 成本估算（新增）

| 操作 | 输入 Token | 输出 Token | 预估成本(DeepSeek) |
|------|-----------|------------|-------------------|
| 意图识别 | ~200 | ~100 | ¥0.0004 |
| 槽位填充 + 追问 | ~500 | ~200 | ¥0.001 |
| 每日播报 | ~800 | ~400 | ¥0.0016 |
| 每日复盘 | ~1000 | ~500 | ¥0.002 |
| 自由对话(10轮) | ~2000 | ~1000 | ¥0.004 |

**日预估成本：¥0.01-0.05**，月成本 ¥0.3-1.5。成本可控，需在 `usage_logs` 中精确记录。

### 4.4 流式输出方案（修订）

采用**分段 IPC 推送**（而非 WebSocket/SSE）：Main process 用 `res.on('data')` 接收 LLM 流式响应，每 50ms 通过 `webContents.send('ai:chatStream', chunk)` 推送一个片段到 Renderer。体验接近真流式，无需额外端口。

## 六、风险与待确认

| 风险 | 等级 | 缓解 |
|------|------|------|
| LLM 响应延迟影响体验 | 🟡 中 | 意图识别用小模型；本地预筛层减少不必要的 LLM 调用；流式输出改善感知延迟 |
| 多轮对话成本高 | 🟡 中 | 限制轮数上限（3轮）；压缩上下文；token 预算管理 |
| 自动决策错误 | 🟡 中 | 所有自动决策附解释 + 一键修改；置信度低时询问；ai_decisions 反馈闭环 |
| 用户隐私顾虑 | 🟢 低 | 仅发送必要上下文；所有数据本地存储；对话历史自动压缩 |
| 离线不可用 | 🟢 低 | 回退到规则引擎 parseInput.js |
| 冷启动记忆效果差 | 🟡 中 | LLM 做首次推断（temperature=0.1），积累数据后切换到统计推断 |

### 已确认问题

1. **默认模型选择**：用户在模型管理页设置"默认对话模型"，场景绑定可覆盖。优先级：场景绑定 > 默认模型
2. **播报触发时机**：手动触发 + 首次打开静默生成可折叠卡片，不做自动弹窗
3. **语音交互深度**：Phase 1-3 只做"语音转文字"（已有 Whisper），不做纯语音对话（需要 TTS + VAD + 打断检测，工作量翻倍）
4. **多设备同步**：不做。个人效率工具单设备足够
