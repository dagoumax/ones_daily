## 一、文档概述

### 1.1 文档目的

本文档定义「个人AI效率管家」V1.0（MVP）的完整功能规格。面向开发者和设计师，提供可执行的功能描述、交互说明和验收标准。

### 1.2 术语表

| 术语 | 定义 |
|------|------|
| **事项（Task）** | 日程中的最小单元，包含标题、时间、优先级、标签等属性 |
| **知识卡片（Knowledge Card）** | 从已完成事项中提炼的结构化知识单元 |
| **图谱（Graph）** | 节点（事项/笔记/卡片）+ 边（关联关系）构成的可视化网络 |
| **双向链接** | Obsidian 式 `[[引用]]`，链接双方互相感知 |
| **场景（Scene）** | 调用大模型的业务场景，如"日程解析"、"知识提炼"、"TTS播报" |
| **模型路由** | 根据场景自动选择对应大模型 |
| **局域网同步** | 桌面端运行时，同网络设备通过浏览器访问 Web 端 |

### 1.3 参考文档

- 需求概要 v0.3（已确认全部9项决策）
- 竞品调研报告（Motion、Obsidian、Todoist、ChatGPT 等）

## 二、产品概述

### 2.1 产品愿景

**一句话**：以 AI 自然语言交互为入口、以日程管理为主线、以 Obsidian 式知识图谱为沉淀载体的个人效率操作系统。

**核心隐喻**：一个住在你电脑里的 AI 秘书——你说，它记；你做，它学；你回顾，它呈现知识网络。

### 2.2 产品目标

| 目标 | 衡量指标 | V1.0 目标值 |
|------|---------|------------|
| 降低事项创建摩擦 | 从想到到创建完成的平均耗时 | < 10 秒（语音方式） |
| 提升日程回顾率 | 周/月视图打开率 | > 80% 活跃天 |
| 知识沉淀自动化 | 人工操作占比 | < 30%（70% 自动完成） |
| 多模型接入灵活性 | 支持模型数量 | ≥ 5 个 |
| 系统稳定性 | 数据丢失事故 | 0 次 |

### 2.3 产品范围

**MVP（V1.0）包含**：
- 7 大核心模块（详见功能全景图）
- Windows 桌面端 + 局域网 Web 端
- 本地 SQLite 存储 + 定时备份

**明确不做**：
- 手机原生 App（V2）
- macOS 支持（V2+）
- 云端同步（V2+）
- 多用户/团队协作
- 插件系统
- 开源

## 五、模块一：AI语音对话创建事项

### 5.1 模块概述

用户通过语音口述创建事项，系统自动将自然语言转化为结构化的日程条目。支持快速一句话创建和深度多轮对话两种模式。

### 5.2 EARS 需求描述

#### REQ-1.1 语音采集（Ubiquitous）

> The system shall 通过系统默认麦克风采集音频，采样率为 16kHz 单声道，并在录音过程中提供实时音量波形反馈。

**说明**：
- 录音触发方式：点击麦克风按钮 或 全局快捷键（默认 `Ctrl+Shift+V`）
- 录音状态显示：静默（无波形）→ 录音中（绿色波形）→ 处理中（旋转动画）→ 完成
- 最长录音时长：60 秒，超时自动停止并处理
- 支持中途取消（点击取消 或 按 `Esc`）

#### REQ-1.2 语音转文字（Event-driven）

> When 用户完成录音，the system shall 首先尝试本地 Whisper.cpp 引擎转写；若本地引擎不可用或转写置信度 < 0.6，则 fallback 至用户配置的 ASR API 模型。

**说明**：
- 本地引擎：Whisper.cpp，使用 small 或 medium 模型（首次启动时自动下载，约 1.5-3GB）
- API fallback：DeepSeek-ASR / GLM-ASR 等
- 转写结果展示在对话区域，支持手动编辑修正

#### REQ-1.3 智能解析（Event-driven）

> When 语音转写文本就绪，the system shall 调用用户配置的"日程解析"场景默认模型，从文本中提取结构化字段：标题、日期时间、时长、地点、参与人、优先级、标签、重复规则。

**解析规则**：

| 字段 | 提取示例 | 默认值 |
|------|---------|--------|
| 标题 | "和张三开会讨论预算" → 标题 | 原始文本前30字 |
| 日期 | "明天"、"下周三"、"3月15号" | 今天 |
| 时间 | "下午3点"、"15:00" | 当前时间+1h |
| 时长 | "开一个小时"、"30分钟" | 60分钟 |
| 地点 | "在3号会议室"、"腾讯会议" | 空 |
| 参与人 | "和张三、李四" | 空 |
| 优先级 | "很重要"、"urgent" → P0 | P2（普通） |
| 标签 | "工作"、"个人"、"学习" | 自动分类 |
| 重复 | "每周一"、"每天" | 不重复 |

#### REQ-1.4 单次快速创建（Event-driven）

> When 解析结果所有必填字段（标题、日期）置信度均 ≥ 0.8，the system shall 直接创建事项并显示成功提示（Toast），无需二次确认。

#### REQ-1.5 对话式创建（State-driven）

> While 解析结果存在必填字段置信度 < 0.8 或用户开启了"始终确认"模式，the system shall 进入对话确认流程，通过文字+语音追问缺失信息，逐项确认后创建。

**对话流程示例**：
```
用户："明天下午开会"
AI："好的，我听到的是'明天下午开会'。请问具体几点？"
用户："3点"
AI："下午3点，记下了。时长大概多久？"
用户："一个小时吧"
AI："明白。需要设置优先级吗？还是普通优先级就可以？"
用户："普通就行"
AI："已创建：明天（7月11日）15:00-16:00 — 开会 [普通优先级]"
```

#### REQ-1.6 语音纠错（Event-driven）

> When 用户对已创建事项发出修改语音指令（如"把明天的会议改到后天下午"），the system shall 识别目标事项、解析修改意图、执行修改并确认。

#### REQ-1.7 批量创建（Optional）

> Where 用户语音中包含批量创建意图（如"下周一到周五每天上午9点站会"），the system shall 解析重复规则并一次性创建所有事项，完成后汇总报告创建数量。

#### REQ-1.8 异常处理（Unwanted）

> If 录音过程中检测到无有效语音输入持续 10 秒，then the system shall 自动停止录音并提示"未检测到语音，请重试"。

> If 语音转写结果为空或完全无法识别，then the system shall 提示用户"未能识别语音内容，请重新说一次或手动输入"。

> If 大模型 API 调用超时（>30秒）或返回错误，then the system shall 显示具体错误信息并保留原始转写文本，允许用户手动编辑后重试或直接保存为纯文本事项。

### 5.3 交互说明

**录音按钮**：
- 位置：主界面左下角固定悬浮按钮（极简圆形，直径 48px）
- 状态：默认（麦克风图标 / 浅灰）→ 悬停（微放大 + 阴影）→ 录音中（红色脉冲动画）→ 处理中（旋转）→ 完成（绿色勾 1.5s 后恢复）

**对话面板**：
- 从底部滑入的抽屉式面板，占屏幕 40% 高度
- 左侧 AI 消息（浅灰背景），右侧用户消息（主题色背景）
- 每条消息下方显示时间戳
- 底部固定输入栏：文字输入框 + 语音按钮 + 发送按钮

**快捷键**：
- `Ctrl+Shift+V`：开始/结束语音录入
- `Ctrl+Shift+Space`：打开对话面板（文字模式）
- `Esc`：关闭对话面板 / 取消录音

## 六、模块二：日程管理

### 6.1 模块概述

提供日/周/月/年多层级视图，支持事项的创建、编辑、删除、拖拽调整，以及标签、优先级和重复规则管理。

### 6.2 EARS 需求描述

#### REQ-2.1 日视图（Ubiquitous）

> The system shall 默认展示日视图，以时间轴形式排列当日事项，左侧显示时间标尺（00:00-23:59），右侧显示事项卡片。

**说明**：
- 当前时间位置显示红色指示线
- 已过期事项（时间已过且未完成）以半透明+红色边框标识
- 已完成事项以灰色+删除线标识
- 事项卡片显示：标题、时间范围、优先级色条、标签圆点
- 点击事项卡片展开详情面板

#### REQ-2.2 周视图（Ubiquitous）

> The system shall 提供周视图，以 7 列网格显示周一至周日的事项分布，每列顶部显示日期和星期。

**说明**：
- 当天列高亮（浅色背景）
- 事项以迷你卡片显示（仅标题+优先级色条）
- 点击事项弹出详情浮层
- 点击空白区域可快速创建该时段事项

#### REQ-2.3 月视图（Event-driven）

> When 用户切换到月视图，the system shall 以月历网格展示当月所有日期，每个日期格内显示最多 3 个事项缩略（超出显示"+N 更多"）。

#### REQ-2.4 年视图（Event-driven）

> When 用户切换到年视图，the system shall 以 12 个月份卡片展示，每月卡片显示该月事项数量的热力图。

#### REQ-2.5 视图切换（Event-driven）

> When 用户点击视图切换按钮或使用快捷键，the system shall 以平滑过渡动画切换视图，过渡时长 ≤ 300ms。

**快捷键**：
- `D`：日视图
- `W`：周视图
- `M`：月视图
- `Y`：年视图
- `T`：回到今天

#### REQ-2.6 事项 CRUD（Event-driven）

> When 用户在任意视图中执行创建/编辑/删除操作，the system shall 立即更新视图并持久化到 SQLite。

**创建方式**：
1. 语音创建（模块一）
2. 点击空白区域快速创建
3. 点击"+"按钮打开创建表单
4. 快捷键 `Ctrl+N`

**编辑方式**：
1. 点击事项卡片 → 展开详情面板 → 编辑
2. 双击事项卡片 → 进入内联编辑模式

**删除方式**：
1. 详情面板中点击删除
2. 右键菜单 → 删除
3. 选中后按 `Delete` 键
4. 所有删除操作需二次确认

#### REQ-2.7 优先级管理（Ubiquitous）

> The system shall 支持 P0（紧急/红色）、P1（重要/橙色）、P2（普通/蓝色）、P3（低优先级/灰色）四级优先级，通过事项卡片左侧色条和颜色透明度区分。

#### REQ-2.8 标签管理（Ubiquitous）

> The system shall 支持自动标签（AI 解析时自动分类）和手动标签（用户创建和分配）。标签以彩色圆点显示，支持按标签筛选视图。

**预设标签分类**：工作、个人、学习、健康、社交、财务、其他

#### REQ-2.9 重复事项（Ubiquitous）

> The system shall 支持以下重复规则：每天、每周（可选周一至周日）、每月（可选日期）、每年、自定义（Cron 表达式简化版）。

**说明**：
- 重复事项在创建时指定重复规则
- 修改单个实例时提示"修改本次"还是"修改所有"
- 删除单个实例时提示"删除本次"还是"删除所有"

#### REQ-2.10 拖拽调整（State-driven）

> While 用户在周视图或日视图中拖拽事项卡片，the system shall 实时显示目标位置的视觉引导线，释放后更新事项时间并持久化。

#### REQ-2.11 搜索与筛选（Event-driven）

> When 用户在搜索框中输入关键词，the system shall 实时（防抖 300ms）返回匹配的事项列表，支持按标签、时间段、优先级、完成状态组合筛选。

**搜索范围**：标题、描述、地点、参与人、标签名

#### REQ-2.12 桌面通知（Event-driven）

> When 事项的提醒时间到达，the system shall 通过 Windows 原生通知推送提醒，点击通知可打开主界面并定位到该事项。

**说明**：
- 默认提前 15 分钟提醒
- 用户可为每个事项自定义提醒时间（5分钟/15分钟/30分钟/1小时/自定义）
- 通知内容：事项标题 + 时间 + 优先级

#### REQ-2.13 异常处理（Unwanted）

> If 用户尝试创建时间与已有事项重叠的事项，then the system shall 显示温和提示（非阻断），允许用户决定是否继续。

> If 用户尝试将事项拖拽到已过去的时段，then the system shall 阻止操作并提示"无法将事项移动到已过去的时间"。

## 七、模块三：早晨语音播报

### 7.1 模块概述

桌面端定时触发，使用 TTS 语音合成播报当日安排。用户可设置播报时间、内容和语音风格。

### 7.2 EARS 需求描述

#### REQ-3.1 定时触发（Ubiquitous）

> The system shall 在用户设定的播报时间（默认 07:30）自动触发播报，无需手动操作。

#### REQ-3.2 播报内容生成（Event-driven）

> When 触发播报时间到达，the system shall 查询当日所有未完成事项，调用"播报文案"场景默认模型组织自然语言播报稿，内容包含：
> - 日期和星期问候
> - 今日事项数量和概述
> - 按时间顺序逐条播报（标题+时间）
> - 昨日未完成事项提醒
> - 高优先级事项特别提醒

**播报稿示例**：
```
早上好，今天是7月11日星期五。
你今天有5个事项安排。
上午9点，团队站会。
上午10点30分，与客户的产品演示。
中午12点，午餐。
下午2点，需求评审会议——这是高优先级事项。
下午4点，周报撰写。
另外，昨天有1个事项未完成：提交报销申请。
祝你今天顺利！
```

#### REQ-3.3 TTS 语音合成（Event-driven）

> When 播报稿生成完毕，the system shall 优先使用 Edge-TTS（免费）进行语音合成和播放；若 Edge-TTS 不可用，fallback 至用户配置的 TTS API 模型。

**说明**：
- 默认语速：1.0x（正常），支持 0.8x/1.0x/1.2x/1.5x
- 默认音色：中文女声（Edge-TTS: zh-CN-XiaoxiaoNeural）
- 播放控制：暂停/继续/跳过/重播

#### REQ-3.4 播报设置（Optional）

> Where 用户打开播报设置，the system shall 提供以下配置项：
> - 播报时间（默认 07:30）
> - 播报范围（仅今日 / 今日+明日预览）
> - 是否包含已完成事项回顾
> - 是否包含天气信息（需配置天气 API）
> - 语速选择
> - TTS 模型选择
> - 是否启用播报

#### REQ-3.5 异常处理（Unwanted）

> If 播报时间到达时系统处于休眠/关机状态，then the system shall 在下次启动时询问是否补播。

> If TTS 合成失败（网络问题或 API 错误），then the system shall 降级为纯文本弹窗显示播报内容。

## 八、模块四：个人知识库与图谱

### 8.1 模块概述

这是产品的核心差异化模块。对标 Obsidian 的知识管理理念，将已完成事项自动归档为知识节点，通过双向链接和可视化图谱呈现知识之间的关联网络。

### 8.2 知识图谱数据模型

#### 节点类型（6种）

| 类型 | 说明 | 来源 |
|------|------|------|
| `task` | 已完成的事项节点 | 日程模块自动创建 |
| `note` | 手动创建的笔记节点 | 用户手动创建 |
| `card` | AI 提炼的知识卡片 | AI 自动或手动触发提炼 |
| `tag` | 标签节点 | 从事项/笔记的标签自动创建 |
| `person` | 人物节点 | 从事项参与人字段提取 |
| `project` | 项目节点 | 用户手动创建或从标签升级 |

#### 边类型（5种）

| 类型 | 说明 | 创建方式 |
|------|------|---------|
| `reference` | 双向链接引用 | 用户在笔记中使用 `[[xxx]]` 语法 |
| `derived_from` | 知识卡片来源 | AI 提炼时自动建立 |
| `tagged` | 标签关联 | 自动 |
| `related` | AI 发现的关联 | AI 推荐 + 用户确认 |
| `manual` | 用户手动关联 | 在图谱中手动连线 |

### 8.3 EARS 需求描述

#### REQ-4.1 自动归档（Event-driven）

> When 用户将事项标记为"已完成"，the system shall 自动将该事项作为 `task` 类型节点存入知识库，保留原始标题、时间、描述、标签、参与人等全部元数据。

#### REQ-4.2 双向链接（Ubiquitous）

> The system shall 支持在笔记中使用 `[[节点名称]]` 语法创建双向链接。链接创建后，双方节点的详情面板均显示对方的引用。

**说明**：
- 输入 `[[` 时自动弹出节点搜索补全面板
- 支持模糊搜索匹配已有节点名称
- 如果 `[[新名称]]` 不存在于已有节点，则自动创建占位节点
- 反向链接自动维护，无需手动管理

#### REQ-4.3 关系图谱可视化（Ubiquitous）

> The system shall 以力导向图（Force-Directed Graph）呈现知识库中所有节点及其关联关系。

**可视化要求**：
- 使用 D3.js 的 `d3-force` 实现力导向布局
- 节点大小反映关联数量（被链接越多越大）
- 节点颜色反映类型（task=蓝, note=绿, card=金, tag=灰, person=紫, project=橙）
- 边的粗细反映关联强度
- 支持以下交互：
  - 拖拽节点重新布局
  - 滚轮缩放（0.1x - 3x）
  - 点击节点选中并高亮其一度关系
  - 双击节点打开详情面板
  - 右键节点显示操作菜单
  - 空白区域拖拽平移画布

#### REQ-4.4 图谱筛选（State-driven）

> While 用户在图谱页面中，the system shall 提供以下筛选维度：
> - 按节点类型筛选（多选）
> - 按标签筛选（多选）
> - 按时间范围筛选（创建时间 / 完成时间）
> - 按关联深度（显示一度/二度/全部关系）
> - 搜索高亮特定节点

#### REQ-4.5 手动建边（Event-driven）

> When 用户在图谱中从一个节点拖拽连线到另一个节点，the system shall 创建 `manual` 类型边，并提示用户选择或输入关系描述。

#### REQ-4.6 AI 关联发现（Event-driven）

> When 新的知识节点（task/card/note）被创建，the system shall 异步调用"知识关联"场景默认模型，分析新节点与已有节点的潜在关联，生成推荐链接列表供用户确认。

**说明**：
- 推荐链接以虚线显示在图谱中，区别于已确认的实线
- 用户可一键确认或忽略
- 确认后的链接转为实线，类型设为 `related`
- 每周定时触发一次全量关联分析（可配置）

#### REQ-4.7 AI 周期总结（Event-driven）

> When 触发总结时间（每日 20:00 / 每周日 20:00 / 每月最后一天 / 每年最后一天），the system shall 调用"知识总结"场景默认模型，生成对应周期的总结报告。

**每日总结结构**：
```
# 每日总结 — 2025年7月10日
- 完成事项：X 项
- 新增知识卡片：Y 张
- 关键成果：...
- 遗留事项：...
```

**每周总结结构**：
```
# 每周总结 — 2025年第28周
- 完成事项：X 项
- 新增知识卡片：Y 张
- 本周关键词：...
- 知识图谱新增关联：Z 条
- 下周重点：...
```

#### REQ-4.8 知识卡片提炼（Event-driven）

> When 用户手动触发对某事项或笔记的"提炼"操作（或 AI 自动识别高价值内容），the system shall 调用"知识提炼"场景默认模型，生成结构化知识卡片。

**知识卡片结构**：
```json
{
  "title": "卡片标题（一句话总结）",
  "content": "详细内容（Markdown）",
  "keywords": ["关键词1", "关键词2"],
  "source_nodes": ["来源节点ID"],
  "related_nodes": ["关联节点ID（AI推荐）"],
  "type": "insight | summary | action | reference"
}
```

#### REQ-4.9 手动笔记（Event-driven）

> When 用户在知识库中创建新笔记，the system shall 提供 Markdown 编辑器，支持以下特性：
> - 实时预览（分屏或切换）
> - `[[双向链接]]` 语法高亮和自动补全
> - 标签 `#tag` 语法
> - 基础 Markdown 语法（标题、列表、粗体、斜体、代码块、引用、表格）
> - 自动保存（防抖 2 秒）

#### REQ-4.10 全文搜索（Event-driven）

> When 用户在知识库搜索框中输入关键词，the system shall 通过 SQLite FTS5 全文索引进行检索，返回匹配的节点列表，按相关度排序。

**搜索范围**：节点标题、节点内容（Markdown 纯文本）、标签名

#### REQ-4.11 回顾与发现（Optional）

> Where 用户打开"发现"页面，the system shall 展示以下智能回顾内容：
> - "去年今天"：展示去年同期的事项和总结
> - "本周关键词云"：本周高频标签和关键词
> - "未连接孤岛"：关联数为 0 的孤立节点列表
> - "潜在关联推荐"：AI 认为可能相关但尚未链接的节点对

#### REQ-4.12 异常处理（Unwanted）

> If 知识图谱节点数超过 5000，then the system shall 默认只渲染一度关联节点，提示用户使用筛选缩小范围以保持性能。

> If AI 关联分析超时或失败，then the system shall 静默失败，不阻塞其他操作，下次定时分析时重试。

## 九、模块五：大模型管理

### 9.1 模块概述

提供统一的大模型配置和管理界面。用户可注册多个模型、按场景绑定默认模型、查看消耗统计。核心设计原则：**模型适配器模式**——新增模型只需实现标准接口。

### 9.2 场景-模型映射

| 场景 | 推荐模型 | 选择理由 |
|------|---------|---------|
| **日程解析** | GLM 5.2 | 中文理解能力强，结构化提取准确 |
| **对话交互** | DeepSeek V3 | 性价比高，中文流畅 |
| **知识总结** | DeepSeek R1 | 推理能力强，适合长文提炼 |
| **知识关联** | DeepSeek V3 | 语义理解好，成本可控 |
| **知识提炼** | Mimo / GLM 5.2 | 结构化输出能力强 |
| **TTS 播报** | Edge-TTS（免费）| 零成本，音质好 |
| **语音识别（ASR）** | Whisper.cpp 本地 | 离线隐私，零成本 |
| **播报文案** | DeepSeek V3 | 轻量文案生成，成本低 |

### 9.3 EARS 需求描述

#### REQ-5.1 模型注册（Event-driven）

> When 用户在模型管理页面点击"添加模型"，the system shall 提供模型注册表单，包含以下字段：
> - 模型名称（用户自定义别名）
> - 模型类型（下拉：LLM / ASR / TTS / Embedding）
> - API Endpoint URL
> - API Key（加密存储）
> - 模型标识（如 `deepseek-chat`、`glm-4`）
> - 额外参数（JSON 格式，可选）

#### REQ-5.2 场景绑定（Event-driven）

> When 用户在场景绑定页面配置，the system shall 为每个场景提供一个下拉选择器，列出已注册的适配模型，用户选择默认模型后保存。

**可配置场景列表**：
1. 日程解析（LLM）
2. 对话交互（LLM）
3. 知识总结（LLM）
4. 知识关联（LLM）
5. 知识提炼（LLM）
6. 播报文案（LLM）
7. 语音识别（ASR）
8. 语音播报（TTS）
9. 文本嵌入（Embedding — 用于语义搜索）

#### REQ-5.3 自动路由（Ubiquitous）

> The system shall 在每次调用大模型时，根据当前场景自动选择用户配置的默认模型，无需用户每次手动选择。

#### REQ-5.4 临时切换（Event-driven）

> When 用户在操作界面（如对话面板）点击模型切换按钮，the system shall 展示可用模型列表，用户选择后仅本次操作使用该模型，不改变默认配置。

#### REQ-5.5 成本统计（Ubiquitous）

> The system shall 记录每次 API 调用的模型、场景、Token 数量（prompt + completion），并按模型和场景维度提供统计视图。

**统计维度**：
- 今日/本周/本月/总计 Token 消耗
- 按模型分布（饼图）
- 按场景分布（柱状图）
- 预估费用（需用户配置各模型价格）
- 费用预警（可设置月度预算上限，达到后提醒）

#### REQ-5.6 连接测试（Event-driven）

> When 用户在模型配置中点击"测试连接"，the system shall 发送一条最小化测试请求，验证 API Key 和 Endpoint 是否可用，5 秒内返回结果。

#### REQ-5.7 异常处理（Unwanted）

> If 模型 API 调用连续失败 3 次，then the system shall 自动将该模型标记为"不可用"状态，后续调用 fallback 至同场景的其他可用模型，并通知用户。

> If 用户未为某场景配置任何模型，then the system shall 在需要调用时提示"请先在设置中配置 [场景名] 的默认模型"。

## 十、模块六：数据管理与备份

### 10.1 模块概述

所有数据存储在本地 SQLite 数据库，支持定时自动备份、手动导出和从备份恢复。

### 10.2 EARS 需求描述

#### REQ-6.1 本地存储（Ubiquitous）

> The system shall 使用 SQLite（better-sqlite3）作为主数据库，数据库文件存储在用户可配置的目录（默认 `%APPDATA%/ai-efficiency/data.db`）。附件和媒体文件存储在数据库同目录下的 `attachments/` 子目录。

#### REQ-6.2 自动备份（Ubiquitous）

> The system shall 每 24 小时自动执行一次数据库备份（默认凌晨 03:00），备份文件命名格式为 `backup_YYYYMMDD_HHmmss.db`，存储在用户配置的备份目录。

**说明**：
- 备份方式：SQLite 在线备份 API（VACUUM INTO），不阻塞正常操作
- 保留策略：默认保留最近 30 个备份，超出自动清理最旧备份（可配置）
- 备份目录建议默认为用户文档目录下的独立文件夹

#### REQ-6.3 手动导出（Event-driven）

> When 用户点击"导出数据"，the system shall 提供以下导出选项：
> - **JSON 格式**：完整数据导出（事项 + 笔记 + 知识卡片 + 图谱结构）
> - **Markdown 格式**：按时间线或标签组织的可读文档
> - **CSV 格式**：仅导出事项列表（兼容 Excel）

#### REQ-6.4 数据恢复（Event-driven）

> When 用户选择备份文件并点击"恢复"，the system shall：
> 1. 先自动创建当前数据的临时备份
> 2. 验证备份文件完整性
> 3. 执行恢复
> 4. 恢复完成后提示用户并重启应用

#### REQ-6.5 存储监控（Optional）

> Where 用户打开存储监控页面，the system shall 展示：
> - 数据库文件大小
> - 附件总大小
> - 各数据表记录数
> - 备份文件数量和占用空间

## 十一、模块七：局域网 Web 同步

### 11.1 模块概述

Electron 主进程内嵌 Express HTTP Server，桌面端运行时，同局域网内的手机/平板/其他电脑可通过浏览器访问 Web 端界面，实时操作同一份数据。

### 11.2 EARS 需求描述

#### REQ-7.1 内嵌 Web Server（Ubiquitous）

> The system shall 在 Electron 主进程启动时同步启动内嵌 Express Server，监听 `0.0.0.0`，默认端口 3000（端口冲突时自动递增）。

**说明**：
- 启动后在桌面端设置页显示局域网访问地址（如 `http://192.168.1.100:3000`）
- 支持生成二维码方便手机扫码访问
- 桌面端退出时自动停止 Web Server

#### REQ-7.2 响应式 Web 界面（Ubiquitous）

> The system shall 提供适配手机/平板屏幕的响应式 Web 界面，精简为以下核心功能：
> - 今日事项列表（日视图简化版）
> - 快速创建事项（文字输入 + 语音按钮）
> - 周视图简化版
> - 知识库搜索和浏览
> - 基础设置

**Web 端与桌面端的关系**：
- Web 端是桌面端的**子集**，不是独立产品
- 桌面端负责所有复杂操作（图谱可视化、模型管理、数据导出等）
- Web 端聚焦移动场景下的查看和快速录入

#### REQ-7.3 实时同步（Event-driven）

> When 用户在 Web 端或桌面端执行任何数据变更操作，the system shall 通过共享 SQLite 数据库实现即时数据一致，另一端在 2 秒内反映变更。

**同步机制**：
- Web 端和桌面端操作同一 SQLite 文件
- 桌面端渲染进程通过 IPC 监听数据库变更事件
- Web 端通过轮询或 WebSocket 接收变更通知（推荐 WebSocket，更实时）

#### REQ-7.4 局域网发现（Optional）

> Where 用户打开"局域网访问"设置，the system shall 显示当前网络 IP 地址、端口号和二维码，并提供可选的安全设置：
> - 访问密码（HTTP Basic Auth）
> - 允许/禁止外网访问（仅局域网）
> - 显示当前连接的设备数

## 十四、数据设计

### 14.1 核心数据表

#### tasks（事项表）

```sql
CREATE TABLE tasks (
  id TEXT PRIMARY KEY,           -- UUID
  title TEXT NOT NULL,           -- 标题
  description TEXT,              -- 详细描述（Markdown）
  status TEXT NOT NULL DEFAULT 'pending', -- pending|in_progress|completed|cancelled
  priority TEXT NOT NULL DEFAULT 'P2',    -- P0|P1|P2|P3
  start_time TEXT,               -- ISO 8601 开始时间
  end_time TEXT,                 -- ISO 8601 结束时间
  location TEXT,                 -- 地点
  participants TEXT,             -- JSON 数组：参与人
  repeat_rule TEXT,              -- JSON：重复规则
  tags TEXT,                     -- JSON 数组：标签
  source TEXT,                   -- 创建来源：voice|manual|batch
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL,
  completed_at TEXT,
  archived_to_kb INTEGER DEFAULT 0  -- 是否已归档到知识库
);
```

#### knowledge_nodes（知识节点表）

```sql
CREATE TABLE knowledge_nodes (
  id TEXT PRIMARY KEY,           -- UUID
  type TEXT NOT NULL,            -- task|note|card|tag|person|project
  title TEXT NOT NULL,
  content TEXT,                  -- Markdown 内容
  metadata TEXT,                 -- JSON：类型特定元数据
  source_task_id TEXT,           -- 来源事项ID（type=task时）
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL,
  FOREIGN KEY (source_task_id) REFERENCES tasks(id)
);
```

#### knowledge_edges（知识边表）

```sql
CREATE TABLE knowledge_edges (
  id TEXT PRIMARY KEY,           -- UUID
  source_id TEXT NOT NULL,       -- 源节点ID
  target_id TEXT NOT NULL,       -- 目标节点ID
  type TEXT NOT NULL,            -- reference|derived_from|tagged|related|manual
  description TEXT,              -- 关系描述（可选）
  weight REAL DEFAULT 1.0,       -- 关联权重
  created_at TEXT NOT NULL,
  FOREIGN KEY (source_id) REFERENCES knowledge_nodes(id),
  FOREIGN KEY (target_id) REFERENCES knowledge_nodes(id),
  UNIQUE(source_id, target_id, type)
);
```

CREATE TABLE model_configs (

## 十六、验收标准

### 16.1 EARS 验收标准示例

#### AC-1：语音创建事项（事件型）

> **When** 用户通过快捷键 `Ctrl+Shift+V` 触发语音录入并说出"明天下午3点和张三开会讨论预算，很重要"，  
> **the system shall** 在 5 秒内完成转写和解析，创建标题为"和张三开会讨论预算"、时间为"明天 15:00-16:00"、优先级为"P0"的事项，  
> **and** 在日视图对应时段显示该事项卡片。

#### AC-2：知识图谱渲染（状态型）

> **While** 知识库中存在 500 个节点和 800 条边，  
> **the system shall** 在 2 秒内完成力导向图初始布局和渲染，  
> **and** 支持 30fps 以上的拖拽和缩放交互。

#### AC-3：早晨播报（事件型）

> **When** 系统时间到达用户设定的播报时间（如 07:30），  
> **the system shall** 自动触发播报流程，在 10 秒内完成播报稿生成和 TTS 合成，  
> **and** 通过系统默认音频输出设备播放，音量为用户设定的系统音量。

#### AC-4：数据备份恢复（事件型）

> **When** 用户选择 3 天前的备份文件并点击"恢复"，  
> **the system shall** 先自动创建当前数据的临时备份，恢复完成后显示成功提示，  
> **and** 重启后事项列表应与备份时间点的状态完全一致。

#### AC-5：双向链接（事件型）

> **When** 用户在笔记 A 中输入 `[[笔记B]]`，  
> **the system shall** 自动创建从笔记 A 到笔记 B 的 `reference` 边，  
> **and** 笔记 B 的详情面板应显示"被笔记 A 引用"的反向链接。

#### AC-6：模型故障切换（异常型）

> **If** 日程解析场景的默认模型连续 3 次调用失败，  
> **then the system shall** 自动标记该模型为不可用，  
> **and** 在下次调用时使用同场景的下一个可用模型，  
> **and** 通过桌面通知告知用户模型切换情况。

## 十八、风险与假设

### 18.1 技术风险

| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|---------|
| Whisper.cpp 在 Windows 上编译/运行兼容性问题 | 语音转写不可用 | 中 | 提供 API fallback，优先测试 Windows 兼容性 |
| D3.js 力导向图在 1000+ 节点时性能下降 | 图谱体验差 | 中 | 实现虚拟化渲染，默认限制渲染范围 |
| SQLite 并发写入冲突（桌面端 + Web 端） | 数据不一致 | 低 | better-sqlite3 默认串行化，单进程访问 |
| Edge-TTS 接口变更或限流 | 播报不可用 | 低 | 提供多个 TTS 选项（ChatTTS 本地备用） |

### 18.2 产品风险

| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|---------|
| 功能过于复杂，MVP 周期过长 | 延迟交付 | 中 | 严格按 P0→P1→P2 优先级交付，P2 可砍 |
| 本地部署导致数据丢失风险 | 用户数据丢失 | 低 | 多重备份机制 + 备份完整性验证 |
| 大模型 API 费用超预期 | 用户成本增加 | 中 | 成本统计 + 预警 + 本地模型优先策略 |

### 18.3 关键假设

| 假设 | 验证方式 |
|------|---------|
| 用户愿意自行申请和配置大模型 API Key | MVP 发布后收集反馈 |
| Whisper.cpp small 模型能在 4GB RAM 设备上流畅运行 | 开发阶段基准测试 |
| 局域网 Web 端使用频率足够支撑开发投入 | MVP 埋点统计 Web 端 DAU |
| 用户对极简风格接受度高于功能密集型设计 | 可用性测试 |
